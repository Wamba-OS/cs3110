# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Vault of Tarnished Sigils** is a Diablo II-themed Magic: The Gathering card catalog web application. It's a full-stack application with a Node.js/Express backend, PostgreSQL database, and vanilla JavaScript frontend (no framework). Card data is sourced from the Scryfall API.

Live app: https://cs3110hub.ddns.net/

## Development Commands

Start the development server:
```bash
npm start
# Server runs at https://localhost:3000 (self-signed cert)
# or at http://0.0.0.0:3000 in production (Render)
```

Install dependencies:
```bash
npm install
```

## Architecture Overview

### Backend (server.js)

**Database Connection**
- Uses PostgreSQL (via `pg` package) hosted on Neon.tech
- Connection string from `DATABASE_URL` environment variable
- Tables auto-created on startup via `initDB()`
- Four tables: `users`, `collection`, `decks`, `deck_cards`
- Foreign key constraints with CASCADE deletes

**Authentication Flow**
- JWT-based auth with 8-hour expiry
- Bcrypt password hashing (10 rounds)
- Admin user auto-seeded from `ADMIN_PASSWORD` env var
- `requireAuth` middleware for protected endpoints
- `requireAdmin` middleware for user creation

**API Structure**
All routes at `/api/`:
- `/api/auth/login`, `/api/auth/register` (POST)
- `/api/collection` (GET public, POST/PUT/DELETE require auth)
- `/api/decks`, `/api/decks/:id`, `/api/decks/:id/cards` (GET public, mutations require auth)
- `/api/wishlist` (GET public, POST/DELETE require auth)

**SSL/HTTPS**
- Production (Render): TLS terminated at edge, app serves plain HTTP
- Local: Self-signed cert generated by `selfsigned` package on first run

### Frontend Architecture

**Shared Client-Side Modules**
All pages load these in order:
1. `utils.js` - Shared utilities (HTML escaping, mana rendering, flash messages)
2. `api.js` - Backend API client wrapper (VaultAPI singleton)
3. `auth.js` - Login modal + nav auth widget (auto-injected on all pages)

**Page-Specific Scripts**
- `cards.js` - Search page (index.html) - Scryfall API integration, grid view, card modal
- `collection.js` - My Vault page (collection.html) - Filter/sort local collection
- `decks.js` - Decks & Wishlist page (decks.html) - Deck builder with inline Scryfall search
- `scan.js` - Scanner page (scan.html) - Tesseract.js OCR for card recognition

**VaultAPI Client Pattern**
The `api.js` module exposes a global `VaultAPI` object with namespaced methods:
- `VaultAPI.auth.login(username, password)` - Returns token, stores in localStorage
- `VaultAPI.auth.logout()` - Clears token, fires 'vault:logout' event
- `VaultAPI.collection.getAll()` - Public endpoint
- `VaultAPI.collection.add(card)` - Requires auth, auto-increments if duplicate
- `VaultAPI.decks.create(data)`, `VaultAPI.decks.addCard(id, card)` - etc.

All authenticated calls include `Authorization: Bearer <token>` header. On 401 response, fires `vault:unauthorized` event which triggers login modal.

**Authentication Integration**
- `auth.js` injects nav widget on DOMContentLoaded
- Widget shows username/role if logged in, or "Login" button
- Admin users see "New User" button to create credentials
- 401 responses auto-trigger login modal via custom event system

### Data Flow Patterns

**Upsert Logic**
Collection and deck card additions check for existing scryfall_id + foil/board combinations and increment quantity instead of creating duplicates:
```javascript
// Example in server.js POST /api/collection
const existing = await pool.query('SELECT * FROM collection WHERE scryfall_id=$1 AND foil=$2', [scryfall_id, foilInt]);
if (existing.rows.length > 0) {
  // Increment quantity
} else {
  // Insert new row
}
```

**Card Type Grouping**
Decks page groups cards by type (Creatures, Instants, Sorceries, etc.) using type_line heuristics in `decks.js:renderCardGroups()`.

**Scryfall API Integration**
- Search page (`cards.js`) uses `/cards/search` endpoint with query builder
- Deck page (`decks.js`) uses inline autocomplete with same endpoint
- Scanner page (`scan.js`) uses Tesseract OCR → fuzzy name match + full-text search
- Wishlist uses `/cards/collection` bulk endpoint to fetch prices

## Database Schema

**users**
- `id` (SERIAL), `username` (UNIQUE), `password_hash`, `role` (admin|author)

**collection**
- Card inventory with `scryfall_id`, `name`, `quantity`, `foil`, `condition`, etc.

**decks**
- Deck metadata: `name`, `description`, `format`, `is_wishlist` (0/1)

**deck_cards**
- FK to `decks.id` ON DELETE CASCADE
- `scryfall_id`, `name`, `quantity`, `board` (main/sideboard)

Wishlist is a special deck with `is_wishlist=1`, auto-created on first access.

## Key Implementation Details

**Mana Cost Rendering**
Mana symbols like `{W}{U}{B}` are rendered as colored pip spans via `utils.js:renderMana()` using CSS classes (`mana-w`, `mana-u`, etc.).

**Card Image Handling**
Scryfall cards may have split faces. Use `utils.js:getImageUri()` and `getField()` helpers to safely extract image_uris and fields from either top-level or card_faces[0].

**Modal Patterns**
All modals use overlay + ESC key + click-outside-to-close pattern. Auth modal is handled by `auth.js`, card detail modal by page-specific scripts.

**Mobile Responsive**
Filter panels slide in as drawers on mobile, controlled by `.drawer-open` class and `#filter-overlay` backdrop.

## Security Notes

- All mutation endpoints require JWT auth via `requireAuth` middleware
- Passwords hashed with bcrypt, never stored as plaintext
- Database connection uses SSL (`rejectUnauthorized: false` for Neon compatibility)
- Static file middleware blocks direct access to `.db`, `.env`, `.pem`, `server.js`, `package.json` files
- HTTPS enforced locally via self-signed cert, TLS terminated at Render edge in production

## CI/CD and Security Scanning

### Automated Security Workflow

The `.github/workflows/security-scan.yml` workflow runs on every PR and push to main, executing three SAST tools:

1. **npm audit** - Scans dependencies for known vulnerabilities
2. **Semgrep** - Static analysis with rulesets: `p/javascript`, `p/nodejs`, `p/express`, `p/security-audit`
3. **Gitleaks** - Secret scanning with full git history

All results are saved to `.security-artifacts/` and uploaded as workflow artifacts (90-day retention).

### Security Scoring

The security-reviewer subagent uses this formula to calculate risk:
```
score = (critical × 40) + (high × 15) + (medium × 5) + (low × 1)
```

**Fail conditions:**
- Any Critical severity finding
- Any High severity finding
- Score ≥ 20

### Local Security Scanning

Run security scans locally before pushing:

```bash
# npm audit only
npm audit

# Full scan (requires Docker for Semgrep, gitleaks binary)
npm audit --json > .security-artifacts/npm-audit.json
semgrep --config=p/javascript --config=p/nodejs --config=p/express --sarif -o .security-artifacts/semgrep.sarif .
gitleaks detect --source . --report-format sarif --report-path .security-artifacts/gitleaks.sarif
```

See `.security-artifacts/README.md` for detailed instructions.

## Automated Security Review (Claude Code Integration)

### Claude Security Review Workflow

The `.github/workflows/claude-review.yml` workflow provides AI-powered security review of every PR:

**Trigger:** Initially `@claude` mention in PR comments, later auto-trigger on PR open/sync

**Process:**
1. Downloads security scan artifacts from security-scan.yml run
2. Invokes the `security-reviewer` subagent (`.claude/agents/security-reviewer.md`)
3. Agent analyzes:
   - npm audit results
   - Semgrep findings (SARIF format)
   - Gitleaks findings (SARIF format)
   - PR diff via `gh pr diff`
4. Returns structured JSON verdict with:
   - PASS/FAIL verdict
   - Risk score (0-100)
   - Specific findings with severity, file, line, CWE, explanation, suggested fix
   - Next steps for implementer
5. Posts formatted comment to PR
6. Adds `security:pass` or `security:fail` label

**Cost Protection:**
- `max_turns: 3` - Prevents runaway loops
- `timeout: 10 minutes` - Workflow timeout
- Expected cost: ~$0.09-0.20 per review
- Budget alert at $5/month in Anthropic Console

**Setup:** See `.github/PHASE4_SETUP_GUIDE.md` for configuration instructions

### Security-Reviewer Agent

The agent (`.claude/agents/security-reviewer.md`) is context-aware of this codebase:
- Knows the Express/PostgreSQL/vanilla JS architecture
- Focuses on high-leverage attack surfaces: SQL injection, auth bypass, XSS, secrets
- Provides specific file:line findings, not generic OWASP advice
- Maps vulnerabilities to actual data flow in this application

**Example finding:**
```json
{
  "severity": "High",
  "title": "Unsanitized user input in innerHTML",
  "file": "decks.js",
  "line": 142,
  "cwe": "CWE-79",
  "source": "manual-review",
  "explanation": "deck.name from user input written to innerHTML without escaping",
  "suggested_fix": "Use escapeHtml(deck.name) from utils.js"
}
```

## Definition of Done

When implementing new features or fixing bugs, a change is considered "done" when:

1. **Code is written** - Implementation follows the architecture patterns documented in this file
2. **Tests pass** - `npm test` runs successfully with no failures
3. **Tests added** - New functionality includes test coverage in the `tests/` directory
4. **Security validated** - No new vulnerabilities in `npm audit`
5. **Code reviewed** - For PR-based workflow: security-reviewer subagent approves
6. **Documentation updated** - User-facing changes documented; architecture changes reflected in this file

### Testing Requirements

- All authentication/authorization changes require tests in `tests/auth.test.js` or `tests/authorization.test.js`
- API endpoint changes require integration tests using Supertest
- Tests must use the test database (via `TEST_DATABASE_URL` env var) to avoid polluting production data
- Run `npm test` before committing; run `npm run test:watch` during development

### Environment Setup for Testing

Create a separate Neon database for testing and add to `.env`:
```
TEST_DATABASE_URL=postgresql://user:password@host/test_db?sslmode=require
TEST_JWT_SECRET=test-jwt-secret
TEST_ADMIN_PASSWORD=test-admin-password
TEST_AUTHOR_PASSWORD=test-author-password
```

If `TEST_DATABASE_URL` is not set, tests will use the production database (not recommended).
